# 物流商辨識規則總覽

本文件列出所有物流商的辨識機制，包含 AI 關鍵字比對、單號格式正則、Track.TW API 支援狀態、以及規則來源。

---

## 辨識流程

```
截圖 → AI (Gemini) 辨識物流商名稱 + 單號
         ↓
   1. CarrierDetector.isValidFormat() → 單號基本格式驗證 (5-50 字元, 英數)
         ↓
   2. AI detectedCarrier 有結果？
      ├── 有 → CarrierDetector 交叉驗證 (confidence ≥ 0.8 且不同 → 以格式為準)
      └── 無 → CarrierDetector.detectBest() 從單號格式判斷
         ↓
   3. 都判斷不了 → 擋掉，顯示錯誤
         ↓
   4. API 驗證 (import + track)
         ↓
   5. 結果無事件且 pending → 擋掉，顯示「查無此單號」
```

## 超商取貨

### 7-ELEVEN 交貨便 (`sevenEleven`)

| 項目 | 內容 |
|------|------|
| Track.TW | 支援 |
| AI 關鍵字 | `7-11`, `7-eleven`, `統一`, `交貨便` |
| 單號格式 | `TW` + 12-15 位數字 + 可選字母 |
| 正則 | `^TW\d{12,15}[A-Z]?$` |
| 信心度 | 0.95 |
| 範例 | `TW259426993523H`, `TW123456789012F` |
| 來源 | Track.TW 實際單號 |

### 全家店到店 (`familyMart`)

| 項目 | 內容 |
|------|------|
| Track.TW | 支援 |
| AI 關鍵字 | `全家`, `familymart`, `family` |
| 單號格式 | 2 字母 + 10-13 位數字 |

| 範例 | `AB1234567890`, `TC1234567890123` |

| 注意 | 全家單號格式較通用，可能與台灣快遞等格式重疊，AI 關鍵字比對更可靠 |

## 純數字格式重疊問題

以下物流商的單號都是純數字，格式互相重疊，**只靠單號無法可靠區分**：

| 碼數 | 物流商 | 信心度 | 備註 |
|------|--------|--------|------|
| 10 碼 | 新竹物流 / 嘉里大榮 / DHL | 0.5 / 0.5 / 0.4 | 全部重疊，需 AI 辨識 |
| 11 碼 | 新竹物流 / 嘉里大榮 / OK超商 | 0.5 / 0.5 / 無規則 | 全部重疊，需 AI 辨識 |
| 12 碼 | 黑貓 / PChome | 0.7 / 0.9 | PChome 以 `12` 開頭可區分 |
| 13 碼 | 中華郵政 | 0.5 | 低信心度 |
| 15 碼 | FedEx | 0.5 | 低信心度 |

**結論：純數字單號必須搭配 AI 從截圖辨識物流商名稱，才能可靠判斷。**
